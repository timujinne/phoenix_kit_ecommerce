20 ## Parsing
21
22 NimbleCSV is by definition restricted in scope to do only
23 parsing (and dumping). The example above discarded the
24 headers when parsing the string, as NimbleCSV expects
25 developers to handle those explicitly later.
26 For example:
27
28 "name\tage\njohn\t27"
29 |> MyParser.parse_string()
30 |> Enum.map(fn [name, age] ->
31 %{name: name, age: String.to_integer(age)}
32 end)
33
34 This is particularly useful with the `c:parse_stream/1` functionality
35 that receives and returns a stream. For example, we can use it
36 to parse files line by line lazily:
37
38 "path/to/csv/file"
39 |> File.stream!(read_ahead: 100_000)
40 |> MyParser.parse_stream()
41 |> Stream.map(fn [name, age] ->
42 %{name: :binary.copy(name), age: String.to_integer(age)}
43 end)
44 |> Stream.run()

46 By default this library ships with two implementations:
47
48 * `NimbleCSV.RFC4180`, which is the most common implementation of
49 CSV parsing/dumping available using comma as separators and
50 double-quote as escape. If you want to use it in your codebase,
51 simply alias it to CSV and enjoy:

56
57 * `NimbleCSV.Spreadsheet`, which uses UTF-16 and is most commonly
58 used by spreadsheet software, such as Excel, Numbers, etc.
59
60 ### Binary references
61
62 One of the reasons behind NimbleCSV performance is that it performs
63 parsing by matching on binaries and extracting those fields as
64 binary references. Therefore, if you have a row such as:
65
66 one,two,three,four,five
67
68 NimbleCSV will return a list of `["one", "two", "three", "four", "five"]`
69 where each element references the original row. For this reason, if
70 you plan to keep the parsed data around in the parsing process or even
71 send it to another process, you must copy the data before doing the transfer,
72 that's why we use `:binary.copy/1` in the examples above.
73
